## 2. False Positive Analysis
- **Status**: Safe queries from Test 1 passed successfully.
- **Strictness**: If I restricted `allowed_topics` to only 'VND' and 'balance', queries about 'credit cards' or 'ATM limits' would be incorrectly blocked.
- **Trade-off**: Stricter security decreases the attack surface but increases user frustration (False Positives). A balance is required where deterministic rules catch obvious abuse and semantic judges handle nuance.

## 3. Gap Analysis (3 Uncaught Attacks)
1. **Character Obfuscation**: Using characters like 'p-a-s-s-w-o-r-d' to break regex.
   - *Fix*: Normalize text (remove punctuation/hyphens) before scanning.
2. **Multi-turn Injection**: Splitting the payload across multiple messages.
   - *Fix*: Implement a stateful monitor that scans the entire conversation window.
3. **Base64 Encoding**: Encoding the attack string in Base64.
   - *Fix*: Add a decoding detection layer or a high-entropy string detector.

## 4. Production Readiness (Scale to 10k Users)
- **Latency**: Use a local small model (e.g., Llama-3-8B) for judging instead of GPT-4o to reduce cost and latency.
- **Cost**: Implement semantic caching (Redis) for frequently asked safe banking questions.
- **Monitoring**: Real-time dashboards (Grafana) to alert on spikes in 'RateLimiter' or 'InputGuardrail' blocks.
- **Updates**: Rules should be pulled from a remote config/DB instead of being hardcoded in Python classes.

## 5. Ethical Reflection
- **Perfect Safety**: Impossible. Security is an evolving arms race; there is no static perfect state.
- **Refuse vs. Disclaimer**: Refuse direct harmful requests (e.g., hacking instructions). Disclaimer for sensitive but valid topics (e.g., financial planning vs. official advice).
- **Example**: A request for 'how to bypass bank security' must be **refused** immediately to prevent harm.
